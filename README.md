# Solana Bot

Интерактивный Python CLI для пакетной работы с Solana-кошельками: генерации и конвертации ключей, проверки балансов, переводов, swap через Jupiter и закрытия пустых token accounts.

> [!WARNING]
> Это инструмент для **Solana mainnet**: операции с активами необратимы. Используйте отдельный кошелёк с небольшой суммой и всегда перепроверяйте адреса, сумму, комиссию и сеть.

> [!CAUTION]
> Вся прикладная логика собрана в одном `bot.py`. Перед работой с реальными средствами всё ещё необходимы тесты, защита ключевого материала и безопасный двухэтапный сценарий подтверждения swap/платежей.

Обычные переводы, swap и операции с SPL требуют обычного system account без данных (`space = 0`) в роли отправителя и fee payer. Durable nonce, stake и program accounts намеренно не используются: бот остановит операцию до подписи и объяснит причину.

## Возможности

В `bot.py` реализован следующий CLI-поток:

| Раздел | Что делает |
| --- | --- |
| Create wallets | Генерирует 12- и 24-словные BIP-39 фразы или Solana keypairs. |
| Converter | Преобразует фразы в ключи и ключи в публичные адреса; доступны стандартная Solana-деривация и legacy-режим. |
| Balance Checker | Проверяет SOL, токены из конфигурации или указанный mint. |
| Multisender | Переводит SOL в режимах «один → много» и «много → один», а также USDC в режиме «один → много» с созданием ATA при необходимости. |
| Swap | Запрашивает котировку и swap-транзакцию у Jupiter. |
| Refund | Ищет и закрывает пустые token accounts, возвращая rent-exempt SOL. |

RPC, Jupiter-клиент, модели и CLI находятся в одном файле; внешний пакет `solana_bot` не требуется.

## Что необходимо исправить до работы с реальными средствами

1. Убрать секретные файлы из Git: `keys.txt`, `phrases.txt`, локальный `config.toml` с реальным API-ключом и при необходимости `history.log`. Один `.gitignore` недостаточен для уже отслеживаемых файлов — их нужно удалить из индекса и проверить историю репозитория.
2. Добавить `config.example.toml`, зафиксированные версии зависимостей и lock-файл.
3. Добавить тесты, прежде всего для расчётов lamports, деривации, подтверждения/повтора транзакций, swap и закрытия token accounts.
4. Расширить проверку полученной Jupiter-транзакции перед подписью: автоматическая preflight-симуляция уже есть, но она не заменяет валидацию ожидаемых инструкций.

## Установка

Минимальная поддерживаемая версия — Python 3.11. Используйте изолированное окружение:

```bash
python3 -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip
python -m pip install -r requirements.txt
python bot.py
```

Прямые runtime-зависимости закреплены в `requirements.txt`. Для полностью воспроизводимой сборки по-прежнему потребуется lock-файл с транзитивными зависимостями, но отдельный `pyproject.toml` в этом проекте намеренно не используется.

## Настройка

Приложение читает [`config.toml`](config.toml). В нём задаются:

- `rpc_url` и `rpc_alternatives` — основной и резервные Solana RPC;
- `api.jupiter_api_url` и `api.jupiter_key` — endpoint и ключ Jupiter для swap;
- `priority_settings`, `default_slippage_bps`, `max_retries`, `confirmation_timeout_sec` и `swap_fee_reserve_lamports`;
- `tokens` — список токенов с mint и decimals;
- `files` — пути к локальным входным данным и истории.

Значение `YOUR_API` — только заглушка. В текущем коде любая непустая строка считается API-ключом, поэтому перед запуском следует добавить явную проверку заглушки и перенести реальный ключ в неотслеживаемое хранилище/переменную окружения. Не коммитьте ключ в `config.toml`.

Пути в секции `files` относительны к текущей рабочей папке. Пока код не переведён на отдельный data directory, запускайте его только из корня проекта и не храните секреты рядом с исходниками.

## Локальные файлы и безопасность

| Файл | Содержимое | Правило |
| --- | --- | --- |
| `phrases.txt` | BIP-39 seed-фразы | Секрет; не добавлять в Git, не отправлять третьим лицам. |
| `keys.txt` | Приватные ключи | Секрет; не добавлять в Git, не хранить в plaintext для рабочих кошельков. |
| `wallets.txt` | Публичные адреса | Не секрет, но требует проверки перед массовыми переводами. |
| `history.log` | Статусы и подписи транзакций | Операционный журнал; может раскрывать финансовую активность. |
| `config.toml` | RPC, токены и потенциально Jupiter API key | Не коммитить реальный ключ; использовать локальный неотслеживаемый конфиг. |

Минимальная мера для временной локальной работы на POSIX-системе — ограничить доступ к секретным файлам владельцем (`chmod 600 …`). Предпочтительное направление развития — encrypted keystore или внешний signer/hardware wallet вместо хранения seed-фраз и ключей в текстовых файлах.

## Безопасный сценарий работы

1. Запустите приложение на devnet/test-кошельке или отдельном mainnet-кошельке с малым балансом.
2. Сначала используйте проверку баланса и убедитесь, что выбран правильный RPC и адреса.
3. Для перевода подготовьте и сохраните неизменяемый список получателей, сравните итоговую сумму и комиссию, затем протестируйте одну небольшую операцию.
4. Для swap бот сначала запрашивает котировку, показывает expected/minimum received и price impact, затем запрашивает отдельное подтверждение для каждого кошелька. Перед broadcast он автоматически симулирует уже подписанные байты; это не публикует транзакцию и не является отдельным режимом dry-run. Не подписывайте непрозрачную сериализованную транзакцию без валидации её инструкций и лимитов.
5. При `pending` или timeout сначала проверьте транзакцию по её подписи в обозревателе, а не запускайте операцию повторно вслепую.

## Проверка качества

В текущем дереве нет каталога `tests`, поэтому команда `python -m unittest discover -s tests -v` намеренно не приводится как рабочая. После появления тестов рекомендуемый минимум:

```bash
python -m unittest discover -s tests -v
ruff check .
python -m pip-audit
```

CI должен устанавливать зависимости из lock-файла, запускать тесты без доступа к mainnet, выполнять линтинг, type-check и secret scan.

## Структура текущей ревизии

```text
SolanaBot/
├── bot.py            # весь CLI, модели, RPC- и Jupiter-клиент
├── config.toml       # локальная конфигурация; не коммитить реальный ключ
├── requirements.txt  # закреплённые прямые runtime-зависимости
├── phrases.txt       # локальные seed-фразы — не должны отслеживаться Git
├── keys.txt          # локальные приватные ключи — не должны отслеживаться Git
├── wallets.txt       # публичные адреса
├── history.log       # журнал операций
└── README.md
```

Для надёжного релиза ещё нужны: тесты, файл игнорирования секретов, lock-файл и CI-конфигурация.

## Приоритетный roadmap

1. Покрыть встроенные RPC- и Jupiter-компоненты unit-тестами с моками.
2. Защитить ключевой материал: `.gitignore`, шаблоны конфигурации, отдельный data directory, права `0600`, затем внешний signer или зашифрованное хранилище.
3. Валидировать инструкции и account keys Jupiter-транзакции перед финальной подписью.
4. Расширить SPL-переводы: decimals и token program уже сверяются on-chain; для Token-2022 mint-ов с расширениями нужен extension-aware builder.
5. Добавить пакетирование инструкций, контролируемую параллельность RPC, экспорт плана платежей в CSV/JSON и возобновление операций по журналу.
